**Authentication Architecture**:

**Access Token (JWT)**:
- Type: JSON Web Token with HS256 signature
- Lifetime: 15 minutes (900 seconds)
- Storage: Client memory (NOT cookies)
- Transmission: `Authorization: Bearer <token>` header
- Payload: `{userId, role, language, iat, exp, nbf}`
- Stateless: Cannot be revoked server-side

**Refresh Token**:
- Type: Random hex string (64 characters)
- Lifetime: 30 days (2592000 seconds)
- Storage: Redis with SHA256 hash key + httpOnly cookie
- Transmission: Automatic via cookie
- Security: Rotation on each refresh
- Revocable: Deleted from Redis on logout

**Security Features**:
- httpOnly cookies prevent XSS attacks
- Secure flag ensures HTTPS-only transmission
- SameSite=Lax prevents CSRF attacks
- Token rotation on refresh prevents replay attacks
- Device tracking (IP + User-Agent)
- Rate limiting on failed login attempts
- Automatic cleanup of expired tokens (Redis TTL)

**Token Flow**:
```
1. POST /auth:login
   → Returns: accessToken (JWT) + refreshToken (cookie)

2. Use accessToken in Authorization header
   → Bearer <accessToken>

3. When accessToken expires (15 min):
   POST /auth:refresh (automatic via cookie)
   → Returns: new accessToken + new refreshToken (rotation)

4. POST /auth:logout
   → Invalidates refresh token in Redis
   → Clears cookie
   → Access token still works until expiry (stateless)
```

**Important Notes**:
- `validateToken` endpoint is internal (localhost only, not tested)
- Passkey authentication (WebAuthn) not tested in this suite
- Cookie persistence may be limited in test environment
- Access tokens work even after logout (until expiry)

```json

  "id": "12345",
  "src_num": "201",
  "dst_num": "202",
  "start": "2025-01-15 10:30:00",
  "answer": "2025-01-15 10:30:05",
```

- `NO ANSWER` - Call was not answered
- `BUSY` - Line was busy
- `FAILED` - Call failed

- Records generated automatically by Asterisk
- Empty list is valid (no calls yet or filtered out)
- Playback endpoint not fully tested (requires existing recordings)

## Common Patterns

### Read-Only Resources
All three endpoints are read-only:
- No POST (create) operations
- No PUT (update) operations
- No PATCH (partial update) operations
- No DELETE operations

### Custom Methods (Google API Design Guide)
All use collection-level custom methods:
- Advice: `:getList`, `:refresh`
- Auth: `:login`, `:refresh`, `:logout`, `:validateToken`
- CDR: `:getActiveCalls`, `:getActiveChannels`, `:playback`

### Response Format
Standard MikoPBX API v3 format:
```json
{
  "result": true,
  "data": [...],
  "messages": {},
  "meta": {
    "timestamp": "2025-01-15T10:30:00+00:00",
    "total": 100,
    "limit": 50,
    "offset": 0
  }
}
```
