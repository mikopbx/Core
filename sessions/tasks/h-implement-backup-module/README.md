# ModuleBackupManager — Модуль резервного копирования MikoPBX

## Problem/Goal

Создать модуль резервного копирования MikoPBX с поддержкой множества удалённых хранилищ, инкрементальных бэкапов записей разговоров, шифрования и уведомлений. Модуль должен обеспечить надёжное сохранение и восстановление всех данных АТС.

**Репозиторий модуля:** `/Volumes/DevDisk/apor/Developement/MikoPBX/Extensions/ModuleBackupManager`
**Имя модуля:** `ModuleBackupManager`

## Транспорты (протоколы хранилищ)

- **WebDAV** — Nextcloud, ownCloud, корпоративные NAS
- **FTP** — legacy-хостинги
- **SFTP** — безопасная передача, стандарт для Linux-серверов
- **S3** — AWS, MinIO, Yandex Object Storage, любой S3-совместимый
- **SMB/NFS** — сетевые шары, NAS (монтируемые директории)

## Категории бэкапа

### 1. Конфигурация АТС + Модули (бэкапятся вместе, восстанавливаются раздельно)
- **mikopbx.db** — через `sqlite3 .dump` (портабельность между версиями PBX)
- **Пользовательские звуковые файлы** (IVR, MOH)
- **Модули:** `/storage/usbdisk1/mikopbx/custom_modules/*/db/module.db` — через `sqlite3 .backup` (большие БД, скорость)
- **Файлы модулей**
- Сертификаты хранятся в mikopbx.db, отдельно не бэкапятся
- **Формат:** tar.gz + metadata.json (версия PBX, дата, состав, размер)
- **Стратегия:** полный бэкап

### 2. История звонков (CDR)
- **cdr.db** — через `sqlite3 .dump`
- **Формат:** tar.gz + metadata.json
- **Стратегия:** полный бэкап

### 3. Записи разговоров
- **Файлы:** `/storage/usbdisk1/mikopbx/voicemailarchive/monitor/*.wav|mp3|webm`
- **Формат:** файлы напрямую (структурированная директория на хранилище)
- **Стратегия:** инкрементальный бэкап
- **Дельта:** по timestamp последнего успешного бэкапа (`ctime > last_backup_timestamp`)
- Хранение `last_backup_timestamp` в локальной БД модуля (per storage + per category)
- При сбое — повтор с предыдущего timestamp

## SQLite бэкап

| БД | Метод | Причина |
|----|-------|---------|
| mikopbx.db | `.dump` (SQL-текст) | Портабельность между версиями PBX, малый размер |
| cdr.db | `.dump` (SQL-текст) | Портабельность, восстановление на другую версию |
| module.db (модули) | `.backup` (бинарная копия) | Скорость для больших БД (гигабайты) |

## Расписание и ротация

- **Расписание:** cron-like конфигурация (ежедневно, еженедельно, произвольное)
- **Ротация:** хранить последние N бэкапов (настраивается per storage per category)
- **Для записей:** ротация не применяется (инкрементальный бэкап)

## Проверка места

- Перед бэкапом оценивать размер данных
- Проверять доступное место на хранилище (если протокол поддерживает)
- Предупреждение при недостатке места

## Шифрование

- Шифрование архива бэкапа (пароль или ключ)
- Бэкап содержит чувствительные данные: пароли SIP-аккаунтов, ключи в mikopbx.db

## Уведомления

- **Email** уведомления при успехе/ошибке бэкапа
- Настраиваемые получатели

## Восстановление

- Восстановление конфига и модулей раздельно
- При восстановлении конфига без модулей — предупреждение пользователю + автоматическое отключение отсутствующих модулей в БД
- Restore доступен через UI и REST API
- Прогресс восстановления через nchan events

## Архитектура

- **StorageTransport** — единый интерфейс для всех транспортов (WebDAV, FTP, SFTP, S3, SMB/NFS)
- **Worker** для фоновых операций бэкапа/восстановления
- **Прогресс-бар** в UI через nchan events (pub/sub)
- **Модуль MikoPBX** — PbxExtensionBase, собственная БД для настроек и состояния
- Отдельный репозиторий в `/Volumes/DevDisk/apor/Developement/MikoPBX/Extensions/ModuleBackupManager`
