# Fix: Attended Transfer Causes Immediate Call Drop

## Problem/Goal

After completing an attended transfer (DTMF ##), the transferred call (SIP-TRUNK ↔ transfer target) drops within 0-1 seconds in some cases. The user reported: "после перевода второго звонка, через секунд 5 сам по себе текущий звонок у пользователя обрывался." On older PBX versions this did not happen.

### Root Cause Analysis (from verbose + AMI logs 2026-02-24)

**Two separate issues identified:**

#### Issue 1: Race condition — `cti_amid_client` AMI Hangup during bridge swap

In failing transfers (C-00000002, C-00000006), `cti_amid_client` sends AMI `Hangup` for the transferor's channel (201) at the exact moment of transfer completion. This does NOT happen in working transfers (C-00000004, C-00000005, C-00000007).

Timeline for C-00000002 (FAILS — call drops in 0.8s):
```
19:38:17.348  AttendedTransfer Result:Success
19:38:17.363  Hangup PJSIP/201 (AMI from cti_amid_client)
19:38:17.983  SIP-TRUNK swapped into bridge with 202
19:38:18.001  transfer_dial_hangup → Hangup(Local/;2)
19:38:18.770  SIP-TRUNK leaves bridge ← CALL DROPS
19:38:18.791  SIP-TRUNK Hangup TechCause:408
```

In `ActionHangupChan.php:154`, when the hangup_chan event for 201 arrives, `StopMixMonitor($row->dst_chan, 'hangupChanEndCalls')` is called. The CDR worker processes the hangup_chan event and takes AMI actions that interfere with the in-progress bridge reorganization.

#### Issue 2: `transfer_dial_hangup` Lua handler kills transferred call

The `h` extension in `internal-users` context checks `ISTRANSFER` and goes to `transfer_dial_hangup`. This handler calls `Hangup()` on both `Local/;2` and then on `PJSIP/202`. When the Local channel pair is destroyed, it may cause a chain reaction that triggers the `Dial()` on SIP-TRUNK's thread to exit, collapsing the bridge.

#### Issue 3: TechCause 408 on ALL calls (systemic)

Every single call in the session (including internal PJSIP/203↔PJSIP/201) terminates with `TechCause:408` (Request Timeout). This suggests a systemic PJSIP or network issue — phones not responding to BYE/re-INVITE in time, or NAT/session timer misconfiguration.

### Key Evidence

- **AMI Hangup correlation**: `Manager 'cti_amid_client' hanging up channel` appears ONLY in failing transfers
- **ActionID `hangupChanEndCalls`**: CDR worker stops MixMonitor on the transfer target during bridge swap
- **`transfer_dial_hangup`**: Calls `Hangup()` which destroys Local channel pair and cascades to bridge teardown
- **TechCause 408**: All calls, including internal, end with 408 — separate systemic issue

### How The Attended Transfer Flow Currently Works

MikoPBX uses a dual-track event system for call detail recording during transfers. Asterisk's Lua dialplan (`extensions.lua`) fires events through CEL (Channel Event Logging) and UserEvents, which are consumed by `WorkerCallEvents` via Beanstalk queue `asterisk-cel`. The PHP-side CDR worker then takes AMI actions (StopMixMonitor, MixMonitor, Hangup) based on channel state in the `CallDetailRecordsTmp` SQLite table.

#### Dialplan Transfer Flow (extensions.conf + Lua)

When user 201 presses `##` (attended transfer, configured in `features.conf` via `FeaturesConf.php`), Asterisk initiates the transfer. The dialplan `internal-users` context (generated by `InternalContexts::generateInternalUsers()` at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Asterisk/Configs/Generators/Extensions/InternalContexts.php:368-419`) contains:

```
exten => _X!,1,Set(CHANNEL(hangup_handler_wipe)=hangup_handler,s,1)
...
same => n,ExecIf($["${DST_CONTACT}x" != "x"]?Dial(${DST_CONTACT},${ringlength},${TRANSFER_OPTIONS}...ekKHhU(${ISTRANSFER}dial_answer)b(dial_create_chan,s,1)))
...
exten => h,1,ExecIf($["${ISTRANSFER}x" != "x"]?Goto(transfer_dial_hangup,${EXTEN},1))
```

The `ISTRANSFER` variable is set to `"transfer_"` when a transfer is in progress. The `Gosub(${ISTRANSFER}dial,${EXTEN},1)` line means it calls `transfer_dial` context instead of `dial` context. The `U(${ISTRANSFER}dial_answer)` means the answer subroutine is `transfer_dial_answer` instead of `dial_answer`.

The critical `h` extension (hangup handler) on line 415: When a channel hangs up in the `internal-users` context and `ISTRANSFER` is set, the dialplan jumps to `transfer_dial_hangup` context. This context is a **Lua** context defined in `extensions.lua` line 1654/1677.

**The same `h` extension pattern exists in multiple contexts:**
- `internal-users` (line 415 in InternalContexts.php) - the primary one for this bug
- `internal` (line 286 in InternalContexts.php)
- `internal-fw` (line 155 in InternalContexts.php)
- `outgoing` (line 85 in OutgoingContext.php)
- `internal-transfer` (line 244 in ExtensionsConf.php) - uses `Goto` not `ExecIf`
- `playback` context (line 216 in ExtensionsConf.php)
- Various provider outgoing contexts (line 296 in OutgoingContext.php)

#### The `event_transfer_dial_hangup()` Lua Function (Issue 2)

Located at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Asterisk/Configs/lua/extensions.lua:1234-1271`:

```lua

    local data = {}
    data['action'] = "transfer_dial_hangup";
    data['end'] = getNowDate()
    data['linkedid'] = get_variable("CHANNEL(linkedid)");
    data['did'] = get_variable("FROM_DID");
    data['agi_channel'] = get_variable("CHANNEL");

    local is_local = string.lower(get_variable("CHANNEL")):find("local/") ~= nil
    if(is_local)then

    if('h' == EXTEN)then

```

**Critical detail:** When called from an `h` extension (hangup handler), `EXTEN` is `"h"`, so `userevent_hangup()` is called. The `userevent_hangup()` function at line 304-313:

```lua
function userevent_hangup(data)
    if(is_test ~= nil) then return end
    data = base64_encode(JSON:encode(data));
    app["CELGenUserEvent"](""..data);
    app["UserEvent"]("CdrConnector,AgiData:"..data);
    app["NoOp"]('Hangup channel ');
    app["Hangup"]();  -- THIS HANGS UP THE CURRENT CHANNEL
end
```

**This is Issue 2's root cause.** When the transferor (201) hangs up their leg after completing the attended transfer, the `h` extension fires in `internal-users`. It goes to `transfer_dial_hangup` Lua context. Since EXTEN is `h`, it calls `userevent_hangup()` which sends the CDR event AND then calls `app["Hangup"]()`. But by this point, the Local channel pair that was bridging the call may still be connected to the SIP-TRUNK channel. The `Hangup()` call on the Local/;2 channel can cascade and destroy the bridge between SIP-TRUNK and the transfer target (202).

In contrast, when called NOT from an `h` extension (e.g., via `Gosub`), `userevent_return()` is used, which calls `app["return"]()` to return to the calling dialplan -- no `Hangup()`.

#### The PHP Worker Side: `ActionHangupChan` (Issue 1)

Located at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Workers/Libs/WorkerCallEvents/ActionHangupChan.php`.

When the `hangup_chan` event arrives via the CEL Beanstalk tube (from the `hangup_handler` context which is set via `CHANNEL(hangup_handler_wipe)=hangup_handler,s,1`), `WorkerCallEvents::callEventsWorker()` processes it via the `otherEvents()` method, which dynamically dispatches to `ActionHangupChan::execute()`.

The `hangupChanEndCalls()` method (line 83-183) does the following:

1. Finds all `CallDetailRecordsTmp` rows matching `linkedid` and `endtime=""` (active calls)
2. For each row where `agi_channel` matches either `src_chan` or `dst_chan`:
   - Sets `endtime` on the CDR row
   - Sets `transfer=0`
3. **Critical line 146-154:** If `src_chan === agi_channel` (the source channel is hanging up), it calls `$worker->StopMixMonitor($row->dst_chan, 'hangupChanEndCalls')` on the **destination** channel.

```php

    $channels[] = ['chan' => $row->src_chan, 'did' => $row->did, 'num' => $row->src_num, 'out' => true];
} else {
    $worker->StopMixMonitor($row->dst_chan, 'hangupChanEndCalls');
    $channels[] = ['chan' => $row->dst_chan, 'did' => $row->did, 'num' => $row->dst_num, 'out' => false];
}
```

The `StopMixMonitor()` method (line 223-238 in `WorkerCallEvents.php`) sends an AMI `StopMixMonitor` command via the `phpagi` AMI user. This is NOT a Hangup command -- it only stops recording. **So Issue 1 is about timing rather than direct Hangup.**

However, after `hangupChanEndCalls`, the code calls `hangupChanCheckSipTrtansfer()` (line 234-291 in `ActionHangupChan.php`). This method:
1. Checks if the hanging up channel is NOT a Local channel
2. Gets the list of active Asterisk channels via AMI `GetChannels`
3. For each paired channel, checks if its `BRIDGEPEER` is in active channels
4. Checks if the `linkedid` has changed (indicating a transfer happened)
5. If so, creates a new CDR row for the SIP transfer and starts a new MixMonitor

**The `cti_amid_client` mentioned in the logs is NOT from the MikoPBX CDR worker.** The MikoPBX AMI user is `phpagi`. The `cti_amid_client` is from an external CTI integration module (like MikoPBX CTI module). This external module sends AMI `Hangup` commands in response to transfer events, causing the race condition described in Issue 1.

#### The `ActionCelAttendedTransfer` Handler

Located at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Workers/Libs/WorkerCallEvents/ActionCelAttendedTransfer.php`.

When a CEL `ATTENDEDTRANSFER` event arrives, this handler:
1. Gets the `Extra` JSON from the CEL event containing `transferee_channel_name` and `channel2_name`
2. Looks up active channel IDs to find linked CDR records
3. If the two linked IDs differ (cross-call transfer), it:
   - Stops MixMonitor on both channels (`$worker->StopMixMonitor($n_data['dst_chan'])` and `$worker->StopMixMonitor($n_data['src_chan'])`)
   - Sets `endtime` on old CDR rows
   - Updates `linkedid` to merge the calls
   - Creates a new CDR row for the transferred call
   - Starts a new MixMonitor on the destination channel

**This happens at the SAME time** as the `hangup_chan` event for the transferor's channel. The race condition is: `ActionCelAttendedTransfer` is processing the transfer merge while `ActionHangupChan` processes the transferor's hangup. Both interact with the same CDR rows and channels via AMI.

#### The `ActionTransferDialHangup` PHP Handler

Located at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Workers/Libs/WorkerCallEvents/ActionTransferDialHangup.php`.

This handles the `transfer_dial_hangup` event sent from the Lua function. It has two branches:

1. **Non-Local channel** (`fillLocalChannelCdr`): Calls `CreateRowTransfer::execute()` to create a new CDR row for the transfer result, then finds CDR rows matching the channel and sets their `endtime`. It also tries to resume MixMonitor recording on the remaining active transfer CDR.

2. **Local channel with no answer** (`fillNotAnsweredCdr`): Handles the case where the transfer consultation call was not answered (transfer was canceled). Sets `endtime` on the unanswered CDR and tries to resume recording.

#### The `CreateRowTransfer` Class

Located at `/Users/nb/PhpstormProjects/mikopbx/Core/src/Core/Workers/Libs/WorkerCallEvents/CreateRowTransfer.php`.

This is called from both `ActionHangupChan::execute()` (line 59) and `ActionTransferDialHangup::fillLocalChannelCdr()` (line 66). It:
1. Checks `$worker->checkChanHangupTransfer` to prevent double processing
2. Finds CDR rows with `transfer=1` for the given `linkedid` and channel
3. If exactly 2 transfer rows exist (normal transfer):
   - Stops MixMonitor on both `src_chan` and `dst_chan`
   - Creates a new CDR row connecting the two remaining parties
   - Starts a new MixMonitor for the new call segment
4. If it's a failed transfer (1 row, unanswered), resumes recording on the original call

### The Race Condition Mechanism

The sequence of events during an attended transfer:

1. User 201 presses `##` to complete the transfer
2. Asterisk fires `ATTENDEDTRANSFER` CEL event
3. Asterisk begins bridge reorganization (moving SIP-TRUNK into bridge with 202)
4. The transferor's channels (201) begin hanging up
5. **Multiple events fire nearly simultaneously:**
   a. `ATTENDEDTRANSFER` CEL -> `ActionCelAttendedTransfer::execute()` -> stops/starts MixMonitor, updates CDR
   b. `hangup_chan` event for 201's channel -> `ActionHangupChan::execute()` -> `StopMixMonitor` on dst_chan, `CreateRowTransfer::execute()`
   c. `transfer_dial_hangup` event (from `h` extension in `internal-users`) -> `userevent_hangup()` -> sends CDR event AND calls `Hangup()` on the Local channel
   d. External CTI module (`cti_amid_client`) may send AMI Hangup for 201's channel

Events (a), (b), and (c) all interact with the same channels and CDR rows. The Local channel `Hangup()` from (c) can cascade to destroy the bridge before Asterisk finishes reorganizing it.

### For Implementation: What Needs to Change

**Issue 2 Fix (Lua `transfer_dial_hangup` calling Hangup on Local channels):**

The `userevent_hangup()` function at line 304-313 calls `app["Hangup"]()` after sending the CDR event. When triggered from the `h` extension during a transfer, this is destructive because:
- The `h` extension handler already runs during channel teardown
- The channel is already being hung up (that's why `h` was triggered)
- Calling `Hangup()` again on a Local channel pair can cascade to other channels still in a bridge

The fix should make the `transfer_dial_hangup` function avoid calling `Hangup()` when it is already in an `h` extension handler context. Alternatively, the `h` extension lines in the PHP-generated dialplan could be changed to use `Gosub` instead of `Goto` so that `userevent_return()` is called instead of `userevent_hangup()`.

Looking at the code, when the `h` extension triggers `Goto(transfer_dial_hangup,${EXTEN},1)`, EXTEN is "h" in that context, so the Lua function takes the `userevent_hangup` path. But the `h` extension is already a hangup handler -- the channel is BEING hung up. There is no need to call `Hangup()` again.

**Issue 1 Fix (StopMixMonitor during bridge swap):**

In `ActionHangupChan::hangupChanEndCalls()`, line 154 calls `StopMixMonitor` on `$row->dst_chan`. During an attended transfer, the `dst_chan` may now be in a new bridge with the transfer target. The `StopMixMonitor` AMI command itself should not cause a hangup, but it is unnecessary since `ActionCelAttendedTransfer` and `CreateRowTransfer` already handle recording transitions.

The fix should check whether a transfer is in progress before calling `StopMixMonitor` on the destination channel. If `$row->transfer === '1'`, the recording management should be deferred to the transfer-specific handlers.

**The `cti_amid_client` Issue:**

This is an external CTI module issue. The `phpagi` AMI user used by MikoPBX's CDR worker does not send Hangup commands during transfer processing. The `cti_amid_client` is a separate AMI connection from a CTI integration module that sends `Hangup` on the transferor's channel at an inopportune time. This may need to be addressed in the CTI module rather than core MikoPBX.
