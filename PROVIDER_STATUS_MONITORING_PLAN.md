# План системы мониторинга статусов провайдеров MikoPBX

## Цель
Простая и эффективная система мониторинга SIP/IAX провайдеров для малого бизнеса (5-20 транков) с real-time обновлениями через WebSocket и историей событий для диагностики.

## Архитектура

### Источники данных (Asterisk 20)

**PJSIP провайдеры:**
```bash
asterisk -rx "pjsip show registrations"      # Статус регистрации
asterisk -rx "pjsip qualify <endpoint>"      # Проверка доступности
```

**IAX2 провайдеры:**
```bash
asterisk -rx "iax2 show registry"           # Статус регистрации
asterisk -rx "iax2 show peer <peer>"        # Детали пира
```

### Хранение в Redis

```
providers:status:{uniqid}         # Текущий статус провайдера (TTL: нет)
providers:history:{uniqid}        # История событий (TTL: 30 дней)
providers:daily:{uniqid}:{date}   # Суточная статистика (TTL: 30 дней)
```

## Структуры данных

### Текущий статус
```json
{
  "uniqid": "SIP-1234567890",
  "type": "sip",
  "state": "registered",        // registered | unregistered | unreachable
  "stateText": "Зарегистрирован",
  "stateColor": "green",        // green | yellow | red
  "host": "sip.provider.com",
  "lastUpdate": 1704888000,
  "lastChange": 1704880000,
  "rtt": 25,                     // Round-trip time в мс
  "uptime": "2ч 15м"            // Человекочитаемое время работы
}
```

### Событие в истории
```json
{
  "timestamp": 1704888000,
  "date": "2024-01-10 14:30:00",
  "type": "error",              // info | warning | error
  "event": "Потеря регистрации",
  "details": "Registration timeout - no response from server",
  "duration": "5 мин",          // Длительность проблемы
  "state": "unregistered"
}
```

### Суточная статистика
```json
{
  "date": "2024-01-10",
  "availability": 99.5,         // Процент доступности
  "downtime": "2м 30с",         // Общее время недоступности
  "incidents": 2,               // Количество инцидентов
  "avgRtt": 28                  // Средняя задержка
}
```

- Расчет uptime и статистики доступности

## Оптимизация для малого бизнеса

### Упрощения:
1. **Без сложной аналитики** - только базовая статистика
2. **Без прогнозирования** - только текущее состояние и история
3. **Без автоматических действий** - только мониторинг
4. **Минимум настроек** - работает "из коробки"

### Оптимизация памяти Redis:
1. **История**: максимум 1000 событий на провайдера
2. **Автоочистка**: данные старше 30 дней удаляются
3. **Компактное хранение**: только необходимые поля
4. **Расчетное потребление**: ~500KB на провайдера

### Частота обновления:
- **Активная страница**: каждые 5 секунд через WebSocket
- **Фоновый режим**: каждые 30 секунд
- **При проблемах**: каждые 10 секунд до восстановления

## Реализация

### Этап 1: Backend
1. Расширить `WorkerProviderStatusMonitor`:
   - Добавить запись событий в Redis
   - Рассчитывать статистику доступности
   - Оптимизировать публикацию в EventBus

2. Создать API endpoints в `ProvidersManagementProcessor`:
   - `getStatusAll()` - статусы всех провайдеров
   - `getHistory($id)` - история событий
   - `getStats($id)` - статистика

### Этап 2: Frontend
1. Обновить `providers-index.js`:
   - Заменить AJAX на WebSocket подписку
   - Добавить real-time индикаторы статуса
   - Визуализировать uptime

2. Создать компонент истории в `provider-modify.js`:
   - Вкладка "Статус и история"
   - Таблица событий
   - График доступности
   - Экспорт отчета

### Этап 3: Интеграция
1. Настроить EventBus каналы
2. Протестировать real-time обновления
3. Оптимизировать производительность

## Примеры использования

### Администратор видит проблему:
1. Открывает список провайдеров
2. Видит красный индикатор у Provider-1
3. Кликает на провайдера
4. Переходит на вкладку "Статус и история"
5. Видит последние события и время начала проблемы
6. Копирует отчет и отправляет провайдеру

### Анализ качества связи:
1. Открывает провайдера
2. Смотрит график доступности за месяц
3. Видит частые сбои в определенное время
4. Экспортирует историю для детального анализа

## Результат

Простая, понятная и эффективная система мониторинга, которая:
- **Визуально понятна** администратору любого уровня
- **Предоставляет необходимую информацию** для диагностики
- **Не перегружена** лишним функционалом
- **Экономна** в использовании ресурсов
- **Полезна** для взаимодействия с техподдержкой провайдера
